# Changelog

All notable changes to this project will be documented in this file. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## [Unreleased]

### Changed
- Surface `callable: false` on `RelationEdge` so consumers know which inverse edges are manifest-only discovery hints vs fully callable joins. davepi M0.5 marks every auto-populated inverse `hasMany` (synthesised from a sibling's `belongsTo`) as `callable: false` because the runtime relation map is per-schema — the inverse never reaches REST `__include`, MCP relation tools, or GraphQL graph edges. `DescribeRelation` adds the `callable?: boolean` field on its `hasMany` and `hasOne` variants; `RelationEdge` surfaces it through `SchemaRegistry.relations()` when present. Existing UI consumers (`<RelatedList>`, `<RelationPicker>`) already fetch children via foreign-key filter on the target list rather than via `__include`, so no behaviour changes today — the flag is the documented contract for future code paths that might naively try `__include=quotes` against an inverse-only edge.
- Trim redundant per-resource overrides in the example app shell. Now that the davepi backend ships `label` / `pluralLabel` / `displayField` natively on every seed schema (projik/davepi#158), the override files at `packages/ui-app-react/src/resources/*.ts` drop the duplicated hints and keep only the consumer-specific surface: `category` (sidebar grouping — UI-only), `listColumns` (table column ordering + overrides — UI-only), `formSections` (form grouping — UI-only), and `actions.bulk` (bulk action wiring — UI-only). Quote, product, category, project shrink to a single-line `category` declaration; account keeps its bulk-delete action + custom column labels; contact keeps its three-section Identity/Contact/Address layout. End-to-end Playwright smoke verifies the trimmed config still renders the full sidebar (CRM / Catalogue / Delivery groups via `category`), auto-discovers the Quotes tab on contact detail via the backend's synthesised inverse, opens the inline RelatedCreateModal pre-stamped with `contactId`, and lands the new quote in the embedded list — zero console / page errors.

- M0.5 follow-up: now that davepi's `/_describe` auto-populates inverse `hasMany` edges from sibling `belongsTo` declarations and ships explicit `label`/`pluralLabel`/`displayField`/`widget`/`format`/`enum`/`stamped` hints (projik/davepi#157), the client-side relation graph and field-hint synthesis in `SchemaRegistry` becomes redundant. Dropped `SchemaRegistry.buildInverseGraph()` and the FK-by-convention sniffing that scanned every `field.reference`/`*Id` name to register synthetic inverses — backend now does it. `SchemaRegistry.relations(path)` returns the manifest's declared edges in order; backend-synthesised inverses surface with `inverse: true` + `declared: false` so consumers can still distinguish them from author-declared edges. Display fallbacks (`name`/`title`/`<path>Name`) remain as defence against backends that haven't shipped explicit `displayField`, but `entry.displayField` always wins. `DescribeField` and `DescribeRelation` types now declare the new backend hints as first-class (no more "planned" comments on the optional fields). Fixture + 9 registry tests updated to match the new shape: backend pre-populates `account.relations.deals` with `inverse: true`, `contact.accountId` is marked `stamped: true` (tenant marker, not a parent FK so no inverse synthesises), and the empty-relations case is exercised explicitly. 36/36 vitest green.

### Added
- M4: agent surface + docs + Storybook. The MCP server in `@davepi/ui-mcp` gains `relation_graph` — a flat edge list across every resource pair, declared `belongsTo`/`hasMany`/`hasOne` plus synthetic inverses from FK-by-convention, deduped on `(source, target, foreignKey, kind)` — so an agent reasoning about parent-child layouts gets the whole map in one call instead of walking `list_resources` per resource. The full tool surface is now six: `list_components`, `describe_component`, `list_recipes`, `validate_page_spec`, `list_resources`, `relation_graph`. A manifest contract test ships in `packages/ui-core/test/manifest.test.ts`: vitest snapshots of `componentManifest` and `recipes` plus invariant checks (every entry has name + category + package + at least one example, every recipe has a unique id). PRs that change the agent-facing surface must update the snapshot deliberately. A Claude Code skill ships at `docs/skill/davepi-ui/SKILL.md` — frontmatter triggers on davepi-ui keywords + project signals (presence of `packages/ui-app-react`, `davepi-ui.config.ts`, `src/resources/`, davepi mentions in CLAUDE.md). The skill body teaches Claude the conceptual model (describe → registry → resolveWidget → page), eight recipe cookbook entries (override list columns, sidebar categories, form sections, JSON-descriptor pages, parent→child detail tab, custom widget, custom field widget, ACL gating), a file-finding cheat sheet, the MCP tool reference, and a don't-dos list (don't store the access token in localStorage, don't fetch `/_describe` more than once, don't bypass the merge order). A `playground/` workspace runs Storybook 8 (`@storybook/react-vite`) auto-discovering stories from `packages/ui-app-react/src/**/*.stories.tsx` plus `playground/stories/`; the build outputs ~890 kB of preview JS and serves on `:6006` via `pnpm --filter @davepi/ui-playground storybook`. An Astro Starlight docs site lives in `docs/` with 14 pages organised into Start / Concepts / Reference / Agent integration. Concepts pages cover the schema-driven data flow, widget resolution rules + override precedence, the relation graph + synthetic inverses, override layers, and ACL gating; Reference covers `defineConfig` + `ResourceConfig` shape, the descriptor JSON schema, and the public component manifest; Agents covers the MCP server tool reference, the curated recipes (list-with-search, detail-with-children, pre-stamped-create), and how to drop the Claude Code skill into a consumer project. Pagefind indexed 13 pages / 766 words.
- M3: override layer + ACL gating + bulk/row actions. Two configuration sources compose on top of the schema-driven defaults: an app-wide `davepi-ui.config.ts` (branding, base URL, sidebar category ordering, app-wide widget overrides) and per-resource override files dropped at `src/resources/{path}.ts(x)` exporting a `ResourceConfig`. The override loader uses Vite's `import.meta.glob` to auto-discover every file in that directory at build time — no registration step, no manifest to maintain. `<ConfigProvider>` exposes the merged config to React; `useResourceConfig(path)` returns the deep-merged effective config for one resource (deep merge with array replacement so consumer `listColumns` deterministically overrides defaults rather than concatenating). `<Sidebar>` now groups resources by `config.category` with optional ordering via `config.categoryOrder`, uses consumer-supplied `label`/`pluralLabel`, and hides resources excluded by `permissions.list` or backend `acl.list`. `<ResourceTable>` honours `listColumns` (string array or full `ColumnSpec` objects with format hints — `currency:USD`, `date`, etc.), surfaces `actions.row` as a Radix DropdownMenu kebab on each row (defaults to Edit + Delete when not configured), and shows a sticky `<BulkActionBar>` with `actions.bulk` when at least one row is selected. The default `bulkDelete` action wires straight to `useDeleteResource` for each id. `<ResourceForm>` renders fields inside `formSections` groups when configured (headings + descriptions), honours `widgets` overrides keyed by field name, and applies field-level ACL: fields with `acl.read` not matched by the JWT roles disappear from the form, fields with `acl.create`/`update` not matched render disabled (server is still the source of truth — the UI just declutters). New ACL hooks shipped from `@davepi/ui-react/acl`: `useResourcePerm(path, op)` for list/create/read/update/delete checks, `useFieldAcl(path, field, op)` for per-field gating. New shadcn primitive: Radix DropdownMenu. New components: `<RowActions>`, `<BulkActionBar>`, `<ConfigProvider>`. The example app shell ships representative override files (`account`/`contact`/`quote`/`product`/`category`/`project`) demonstrating the full surface — labels, categories, list columns, form sections (Identity/Contact/Address on contact), and bulk-delete on account.
- M2: relation-aware widgets — the flagship surface that addresses the original Refine pain points. `<RelationPicker>` is now a real Combobox: opens a popover-backed `Command` (cmdk) list, debounces the user's input by 250ms and dispatches `__q` against the target resource list, displays each result via `SchemaRegistry.preview()` (uses the resolved `displayField`), and rehydrates the selected id on mount via `useResource` so the trigger shows the human label even after a page reload — not the raw id. A trailing "+ Create new <Target>" item opens a `<RelatedCreateModal>` so a missing referent doesn't break the user's flow. `<MultiRelationPicker>` ships the same combobox over `[String]` fields with selections rendered as removable chips. `<RelatedList>` embeds a `<ResourceTable>` filtered by the parent foreign key with a header `+ New <Child>` button that opens `<RelatedCreateModal>` pre-stamped with the parent FK. `<ResourceDetailPage>` replaces the M1 link-out buttons with a `<Tabs>` layout: one tab per declared `hasMany`/`hasOne` plus every synthetic inverse that `SchemaRegistry` derives from FK-by-convention. Tabs only render when at least one child relation exists; single-relation parents stay flat. `<ResourceTable>` gained an `embedded` mode (hides the page header so it sits inside `<RelatedList>`) and now picks up filter params from the URL (`?accountId=xxx`) so a child list deep-linked from a parent detail works standalone. The standalone list's "New" button carries active filters through as `?prefill_<fk>=<id>` so a child created from a filtered list inherits the parent FK. `zodFromDescribe` gained `includeStampedFields: string[]` so a hidden parent FK can be unblocked without dragging every server-stamped field into the required schema — without this, a hidden `accountId` silently rejected submit because `userId` came along for the ride. New shadcn primitives: `Popover`, `Command` (cmdk), `Tabs`.
- M1: schema-driven `<ResourceTable>`, `<ResourceForm>` (react-hook-form + zod), and `<ResourceDetail>` ship as the first real component surface. The table reads `/_describe`, picks the first non-stamped fields as default columns, supports sortable headers (`__sort`), full-text search on schemas that declare searchable fields (`__q`), and `__page` pagination. The form auto-generates one widget per describe field via `resolveWidget`, validates with `zodFromDescribe`, supports hidden+readonly pre-stamped fields via `?prefill_<fieldName>=<value>` URL params (so a parent's detail page can hand a child the FK without users typing it), and surfaces server errors at the form root. Detail renders fields in describe order with type-aware formatting, ships Edit/Delete actions (delete behind a shadcn `Dialog` confirm), and links out to every related resource computed from the bidirectional relation graph. Replaces the M0 stub list/detail pages — the same routes now serve real CRUD.
- M1 widget components for every `WidgetKind` resolved by `@davepi/ui-core`: `TextInput`, `TextArea`, `NumberInput`, `Switch`, `DateInput`, `EnumSelect`, `TagInput` (chip UI with Enter/comma to commit + Backspace to pop), `EmailInput`, `UrlInput`, `CurrencyInput` (reads `spec.currency` from the `currency:USD` format hint), and `JsonEditor` (textarea-backed, parses on blur). `RelationPicker`/`MultiRelationPicker`/`FileUploader` ship as M1 stubs — full picker UX with autocomplete and inline create lands in M2. New shadcn primitives added to support widgets: `Select` (Radix Combobox), `Switch`, `Checkbox`, `Textarea`, `Table`, `Dialog`.
- M1 component manifest in `@davepi/ui-core` and MCP server in `@davepi/ui-mcp`. The manifest hand-curates one entry per published component (props JSON Schema, canonical examples in both spec + JSX form, agent notes); a `react-docgen-typescript` generator will merge auto-derived prop schemas on top in M4. The MCP server exposes five tools over stdio — `list_components`, `describe_component`, `list_recipes`, `validate_page_spec` (runs the input through `PageSpec.parse` and returns parsed value or structured zod issues), and `list_resources` (when `DAVEPI_API_URL` is set, fetches the live `/_describe` and returns paths + display info + the inferred relation graph) — so an AI agent inside Claude Code can browse the framework's vocabulary and emit JSON descriptors that the runtime renders directly. Curated recipes ship (`list-with-search`, `detail-with-children`, `pre-stamped-create`) as known-good patterns agents can lift.
- Initial monorepo skeleton with pnpm + turborepo. Workspaces for `@davepi/ui-core`, `@davepi/ui-react`, `@davepi/ui-app-react`, `@davepi/ui-cli`, `@davepi/ui-mcp`, plus `examples/crm`, `docs`, `playground`. Establishes the M0 foundation: schema-driven default views auto-generated from davepi's `/_describe`, with later milestones layering shadcn-based widgets, relation-aware pickers, per-resource overrides, and an MCP-discoverable component manifest.
